# Harness Integration Spec

> 基于 Anthropic & OpenAI Harness Engineering 思想设计
> 日期：2026-04-02
> 状态：草稿

## 问题背景

当前 xiaoxi-dreams 的 Dream 是单一 agent 执行，存在以下问题：

1. **Self-Evaluation 问题**：Agent 倾向于赞扬自己的工作，即使平庸
2. **Context 膨胀**：所有记忆全量加载，context 稀缺
3. **无外部反馈**：缺乏独立的评审机制
4. **记忆碎片化**：缺乏结构化的组织

## 核心思想

### Anthropic: Generator-Evaluator 架构

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Generator-Evaluator 架构                                │
└─────────────────────────────────────────────────────────────────────────────┘

  Generator (Dream) ──► 输出 ──► Evaluator ──► 反馈 ──► Generator (循环)
                        │                           │
                        │                           │
                        ▼                           ▼
                    记忆报告                    评分 + 批评
```

**关键洞察**：
- 分离 Generator 和 Evaluator 避免自我偏袒
- 调教 Evaluator 比让 Generator 自我批评容易得多
- Evaluator 需要具体可执行的反馈

### OpenAI: Map not Manual

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Map not Manual 思想                                │
└─────────────────────────────────────────────────────────────────────────────┘

  ❌ 错误做法：                          ✅ 正确做法：
  ┌─────────────────────┐                ┌─────────────────────┐
  │   MEMORY.md       │                │   MEMORY.md       │
  │   (1000+ 行)      │                │   (目录索引，~50行) │
  │                   │                ├─────────────────────┤
  │   塞满所有信息    │                │   memory/          │
  │   难以维护        │                │   ├── lessons/    │
  │   信息过载        │                │   ├── decisions/  │
  └─────────────────────┘                │   ├── people/    │
                                          │   └── ...       │
                                          └─────────────────────┘

  Context = 稀缺资源
  给 Agent 一个地图，不是一本说明书
```

### 记忆分层设计 (Map not Manual)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         记忆分层架构                                      │
└─────────────────────────────────────────────────────────────────────────────┘

  Layer 5: MEMORY.md (索引)
  ─────────────────────────────────────────────────────────────────────────
  格式：
  ## [mem-001] 标题
  - 一句话摘要
  - 类型: fact/decision/lesson/procedure/person/project
  - 来源: YYYY-MM-DD
  - 标签: #tag1 #tag2
  
  限制：≤ 100 行，~150 chars/条目

  Layer 4: memory/ 目录
  ─────────────────────────────────────────────────────────────────────────
  memory/
  ├── lessons/        # 教训详情 (长文本)
  ├── decisions/      # 决策详情 (上下文+理由)
  ├── people/         # 人物详情
  ├── procedures/     # 流程详情
  └── projects/      # 项目详情

  Layer 3: .beads/dolt (结构化存储)
  ─────────────────────────────────────────────────────────────────────────
  完整元数据，可查询
```

### Golden Principles (不可违反规则)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      Golden Principles                                   │
└─────────────────────────────────────────────────────────────────────────────┘

  1. 永不删除原始日志
  2. 永不移除 ⚠️ PERMANENT 标记
  3. 变更超过 30% 必须备份
  4. Evaluator 评分 < 60 分必须重做
  5. Context 超过 50% 必须压缩

  这些规则编码进 Evaluator 的评审标准
```

### Evaluator 评审标准

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      Evaluator 评审标准                                  │
└─────────────────────────────────────────────────────────────────────────────┘

  1. 完整性 (25%)
  ─────────────────────────────────────────────────────────────────────────
  • 是否遗漏重要记忆
  • 是否有明显空白
  • 是否有矛盾信息

  2. 准确性 (25%)
  ─────────────────────────────────────────────────────────────────────────
  • 记忆内容是否正确
  • 分类是否合理
  • 重要性评分是否恰当

  3. 可读性 (20%)
  ─────────────────────────────────────────────────────────────────────────
  • 摘要是否清晰
  • 结构是否合理
  • 标签是否准确

  4. 连贯性 (15%)
  ─────────────────────────────────────────────────────────────────────────
  • 与已有记忆是否一致
  • 是否有逻辑矛盾
  • 关联是否正确

  5. 可操作性 (15%)
  ─────────────────────────────────────────────────────────────────────────
  • 未来能否快速检索
  • 能否快速理解上下文
  • 是否有足够的行动指引

  总分 < 60 → 返回 Generator 重做
  总分 60-80 → 返回 Generator 改进
  总分 > 80 → 通过
```

## 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Evaluator 过度严格 | 循环次数过多 | 设置最大迭代次数 |
| Token 消耗增加 | 成本上升 | 优化 prompt，使用便宜模型做 Evaluator |
| 实现复杂度高 | 开发周期长 | 分阶段实现，优先 MVP |
